# Методология исследования
## Лабораторная работа №6: Тестирование и оценка качества прототипа ProjectFlow

---

## 1. Общие положения

### 1.1 Цель исследования

Провести комплексную оценку качества пользовательского интерфейса системы управления проектами **ProjectFlow** с использованием современных методов usability-тестирования для выявления проблем взаимодействия и формирования рекомендаций по улучшению пользовательского опыта.

### 1.2 Задачи исследования

1. Оценить информационную архитектуру системы методом обратной карточной сортировки
2. Проверить соответствие интерфейса ожиданиям пользователей
3. Провести экспертную оценку по чеклистам usability
4. Количественно оценить эффективность форм методами GOMS и KLM
5. Измерить производительность пользователей при выполнении типовых задач
6. Определить общую удовлетворенность системой по шкале SUS
7. Сформировать приоритизированный список рекомендаций по улучшению

### 1.3 Объект исследования

**ProjectFlow** - веб и мобильное приложение для управления проектами с поддержкой методологий Agile, Scrum, Kanban.

**Платформы:**
- Web-приложение (desktop, 1920×1080)
- Мобильное приложение (iOS/Android, 5-7")

## 2. Методы исследования

### 2.1 Обратная карточная сортировка (Reverse Card Sorting)

**Цель:** Проверить понятность структуры навигации для пользователей.

**Методика:**

**Оффлайн тест:**
- Участникам предоставляется текстовое дерево навигации
- Даются карточки с описанием задач (30 карточек)
- Участники указывают, в каком разделе они найдут нужную функцию
- Фиксируется процент правильных попаданий и ошибочных путей

**TreeJack тест (моделирование):**
- Создается онлайн-дерево в формате TreeJack
- 10 виртуальных участников проходят 15 заданий
- Анализируются метрики: успешность, время, косвенность путей

**Критерии успеха:**
- ≥80% правильных попаданий в ключевые разделы
- ≥70% для второстепенных разделов
- Косвенность (directness) ≥75%

### 2.2 Тестирование ожиданий (Expectation Testing)

**Цель:** Выявить несоответствия между ожиданиями пользователей и реальным поведением системы.

**Методика:**
- Отбирается 20 критичных элементов интерфейса
- Для каждого формулируется вопрос: "Что произойдет, если...?"
- Ответы сравниваются с запланированным поведением
- Выявляются элементы с неочевидным поведением

**Критерии успеха:**
- ≥85% совпадений ожиданий с реальностью для критичных элементов
- ≥70% для некритичных элементов

### 2.3 Оценка по контрольным спискам (Checklist Evaluation)

**Цель:** Экспертная оценка соответствия интерфейса стандартам usability.

**Методика:**

**Веб-приложение:**
- Чеклист из 150 пунктов по категориям:
  - Архитектура и навигация (25 пунктов)
  - Планировка и дизайн (30 пунктов)
  - Содержание (20 пунктов)
  - Формы и взаимодействие (25 пунктов)
  - Визуальный дизайн (25 пунктов)
  - Устойчивость к ошибкам (15 пунктов)
  - Производительность (10 пунктов)

**Мобильное приложение:**
- Чеклист из 100 пунктов:
  - Навигация и архитектура (20 пунктов)
  - Жесты и взаимодействие (25 пунктов)
  - Адаптивность (15 пунктов)
  - Визуальный дизайн (20 пунктов)
  - Производительность (10 пунктов)
  - Платформенные стандарты (10 пунктов)

**Оценка:** Каждый пункт: Да (1) / Нет (0) / Не применимо (-)

**Критерии успеха:**
- ≥85% положительных оценок - отличный интерфейс
- 70-84% - хороший, требует улучшений
- <70% - требует серьезной доработки

### 2.4 GOMS анализ веб-интерфейса

**Цель:** Количественная оценка эффективности форм ввода.

**Методика:**
- Анализируются 3 ключевые формы
- Для каждой формы создаются 2 варианта дизайна
- Каждый вариант оценивается для 2 методов ввода
- **Итого: 12 моделей расчета**

**Операторы GOMS:**
- K = 0.2 с (нажатие клавиши)
- P = 1.1 с (перемещение указателя мыши)
- H = 0.4 с (перемещение руки клавиатура↔мышь)
- M = 1.35 с (ментальная пауза)

**Формы для анализа:**
1. Создание проекта (8-10 полей)
2. Создание задачи (6-8 полей)
3. Управление пользователем (7-9 полей)

**Методы ввода:**
- Метод A: Навигация Tab + Enter (клавиатура)
- Метод B: Навигация мышью + клики

**Критерий:** Выбирается вариант с минимальным временем выполнения.

### 2.5 KLM/TLM анализ мобильного интерфейса

**Цель:** Оценка эффективности жестового управления.

**Методика:**
- Анализируются 3 типовых сценария
- Используется Touch Level Model (TLM)

**Операторы TLM:**
- T (Tap) = 0.3 с (нажатие на элемент)
- P (Point) = 0.4 с (перемещение пальца)
- D (Drag) = 0.6 с (перетаскивание)
- F (Flick) = 0.4 с (смахивание)
- M = 1.2 с (ментальная пауза)

**Сценарии:**
1. Создание задачи (мобильная форма)
2. Изменение статуса задачи (drag & drop)
3. Просмотр деталей проекта (навигация)

### 2.6 Usability-тестирование

**Цель:** Измерить реальную производительность пользователей.

**Участники:** 4 человека, соответствующих персонажам:
- Анна (PM, 32 года) - опытный пользователь
- Дмитрий (Dev, 28 лет) - технически подкован
- Игорь (Стажер, 22 года) - новичок
- Сергей Александрович (Руководитель, 45 лет) - нечастый пользователь

**Задания:** 7 тестовых сценариев различной сложности

**Метрики:**
- **Task Success** - процент успешного завершения
- **Time-on-Task** - время выполнения (среднее, медиана, разброс)
- **Errors** - количество и тип ошибок
- **Efficiency** - количество шагов/кликов

**Целевые показатели:**
- Успешность: ≥80% для критичных задач
- Время: в пределах установленных критериев
- Ошибки: ≤2 на задачу

### 2.7 SUS анкетирование

**Цель:** Определить общую удовлетворенность системой.

**Методика:**
- Стандартная SUS анкета (10 вопросов)
- Шкала Ликерта 1-5
- Проводится после usability-тестирования

**Расчет:**
```
SUS Score = (Σ нечетных - 5) + (25 - Σ четных) × 2.5
```

**Интерпретация:**
- ≥80 - отличный интерфейс
- 68-79 - хороший (выше среднего)
- 51-67 - средний
- <51 - проблемный

**Цель для ProjectFlow:** ≥75 баллов

## 3. Моделирование данных

### 3.1 Принципы моделирования

Поскольку реальные пользователи недоступны на момент выполнения работы, данные моделируются на основе:

1. **Персонажей из ЛР №1** с детальными профилями
2. **Стандартных паттернов** поведения (Nielsen Norman Group)
3. **Статистики** для систем управления проектами
4. **Логики сценариев** и сложности задач

### 3.2 Модель поведения участников

**Анна (PM, опытный пользователь):**
- Время выполнения: -30% от среднего
- Успешность: 90-100%
- Ошибки: минимальные
- SUS оценка: 75-85

**Дмитрий (Dev, средний уровень):**
- Время выполнения: ±10% от среднего
- Успешность: 80-90%
- Ошибки: 1-2 на сложную задачу
- SUS оценка: 70-80

**Игорь (Стажер, новичок):**
- Время выполнения: +40% от среднего
- Успешность: 60-80%
- Ошибки: 2-4 на задачу
- SUS оценка: 60-70

**Сергей (Руководитель, нечастый пользователь):**
- Время выполнения: +20% от среднего
- Успешность: 70-85%
- Ошибки: 1-3 на задачу
- SUS оценка: 65-75

### 3.3 Распределение успешности

**По сложности задач:**
- Простые (уровень 1): 90-100% успех
- Средние (уровень 2): 75-90% успех
- Сложные (уровень 3): 60-80% успех

**Разброс времени:** ±15% от базового значения (нормальное распределение)

### 3.4 Паттерны ошибок

**Типичные ошибки:**
- Навигационные: неверный раздел меню (15-20%)
- Заполнение форм: пропуск обязательных полей (10-15%)
- Ожидания: непонимание результата действия (5-10%)
- Поиск: неэффективные запросы (10-12%)
