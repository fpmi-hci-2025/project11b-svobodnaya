# Лабораторная работа №6
## Тестирование и оценка качества прототипа приложения

**Проект:** ProjectFlow - Система управления проектами  
**Платформы:** Веб-приложение и мобильное приложение  
**Дата выполнения:** 07.11.2025

---

## Цель работы

Провести комплексную оценку качества пользовательского интерфейса системы ProjectFlow с использованием различных методов тестирования для выявления проблем юзабилити и формирования рекомендаций по улучшению интерфейса.

---

## Выполненные задания

### Задание 1. Определение целей тестирования и численных критериев качества

**Выполнено:**

Разработано 7 тестовых заданий для оценки интерфейса:
1. Создание нового проекта
2. Создание новой задачи
3. Изменение статуса задачи
4. Поиск и фильтрация задач
5. Просмотр аналитики проекта
6. Управление пользователями (администрирование)
7. Создание команды

Для каждого задания определены численные критерии:
- Целевой процент успешного выполнения (75-90%)
- Целевое время выполнения (15-75 секунд в зависимости от задачи)
- Допустимое количество ошибок (0-1 на задачу)

**Документация:** `01_test_tasks.md`, `02_quality_criteria.md`

### Задание 2. Проведение тестирования

#### 2.1. Обратная карточная сортировка

**Метод:** Tree Testing (оффлайн формат)

**Объем работы:**
- Разработано дерево навигации (8 основных разделов, ~40 подразделов)
- Создано 30 тестовых карточек
- Проведено тестирование с 5 участниками

**Результаты:**
- Общая успешность: 74.7%
- Проблемные категории: Команда (60%), Настройки (64%)
- Выявлено 4 карточки с критически низкой находимостью (<50%)

**Документация:** `07_card_sorting/`

---

#### 2.2. Тестирование ожиданий

**Метод:** Expectation Testing

**Объем работы:**
- Разработано 10 тестовых сценариев
- Протестировано 4 участника
- Измерено совпадение ожиданий пользователей с фактическим поведением системы

**Результаты:**
- Общее совпадение ожиданий: 76.25%
- Выявлено 2 критических несоответствия (12.5% совпадение):
  - Применение фильтров (ожидается мгновенное, требуется кнопка)
  - Функция создания команды (неочевидное размещение)
- Стандартные паттерны: 100% узнаваемость

**Документация:** `10_expectation_testing/expectation_test.md`

#### 2.3. Исследование прототипа по чеклистам

**Веб-приложение:**

**Мобильное приложение:**

**Документация:** `03_checklist_web.md`, `04_checklist_mobile.md`

#### 2.4. Анализ интерфейса по модели GOMS

**Метод:** GOMS (Goals, Operators, Methods, Selection rules)

**Объем работы:**
- Проанализировано 3 веб-формы
- Для каждой формы рассчитано 4 варианта взаимодействия
- Итого: 12 моделей

**Формы:**
1. Создание проекта (6 полей)
2. Создание задачи (8 полей)
3. Управление пользователями (10 полей)

**Варианты взаимодействия:**
- Mouse-only
- Keyboard-only
- Mixed (Tab navigation)
- Mixed (Enter jumps)

**Результаты:**
- Keyboard-only на 30-32% быстрее mouse-only для всех форм
- Самая долгая операция: управление пользователями mouse-only (57.6 сек)
- Самая быстрая: создание проекта keyboard-only (28.5 сек)

**Документация:** `05_goms_analysis/`

---

#### 2.5. Анализ интерфейса по модели KLM/TLM

**Метод:** Keystroke-Level Model / Touch Level Model

**Объем работы:**
- Проанализировано 3 мобильных сценария
- Рассчитано время выполнения с учетом операций: Touch (T), Point (P), Drag (D), Flick (F)

**Сценарии:**
1. Создание задачи на мобильном (18.95 сек, 21 операция)
2. Изменение статуса через Drag & Drop (6.20 сек, 8 операций)
3. Просмотр проекта (13.10 сек, 17 операций)

**Результаты:**
- Drag & Drop для смены статуса - наиболее эффективная операция (6.2 сек)
- Все сценарии укладываются в приемлемое время (<20 сек)

**Документация:** `06_klm_analysis/`

#### 2.6. Анализ интерфейса с использованием метрик производительности

**Метод:** Модерируемое usability-тестирование с протоколом "думай вслух"

**Объем работы:**
- 4 участника
- 7 задач на 2 платформах
- Итого: 56 тестовых сессий

**Метрики:**
- Task Success Rate (процент успешного выполнения)
- Time-on-Task (время выполнения)
- Error Rate (количество ошибок)

**Результаты веб-приложения:**
- Средняя успешность: 67.9%
- Среднее время: 54.7 сек
- Средние ошибки: 1.0 на задачу
- Задачи с критическими проблемами: Z4 (50%), Z6 (50%), Z7 (25%)

**Результаты мобильного приложения:**
- Средняя успешность: 50%
- Среднее время: 72.9 сек (+33% относительно веб)
- Средние ошибки: 1.32 на задачу (+32% относительно веб)
- Задачи с критическими проблемами: Z4 (25%), Z6 (0%), Z7 (0%)

**Документация:** `08_usability_testing/test_results.md`

### Задание 3. Анализ результатов

**Выполнено:**

Проведен комплексный анализ результатов всех методов тестирования с выявлением:

**Критические проблемы (приоритет P0):**
1. Механизм фильтрации задач
   - Успешность: 50% (веб), 25% (мобильное)
   - Проблема: отсутствие мгновенного применения фильтров
   
2. Функция "Создать команду"
   - Успешность: 25% (веб), 0% (мобильное)
   - Проблема: неочевидное размещение в навигации
   
3. Управление пользователями
   - Успешность: 50% (веб), 0% (мобильное)
   - Проблема: формы не адаптированы для мобильных устройств

**Высокоприоритетные проблемы (приоритет P1):**
- Отсутствие onboarding для новых пользователей
- Недостаточная оптимизация мобильных форм
- Недостаточная обратная связь системы
- Несогласованность элементов интерфейса

**Сильные стороны:**
- Kanban доска: 100% успешность
- Базовые операции создания проектов/задач: 75-100%
- Соответствие стандартам по чеклистам: 90%+
- Стандартные паттерны взаимодействия

**Корреляция метрик:**
- SUS балл сильно коррелирует с фактической производительностью (r = 0.95)
- Тестирование ожиданий подтверждает результаты usability-тестирования
- Проблемы, выявленные разными методами, согласуются между собой

### Задание 4. Формирование рекомендаций

**Выполнено:**

Разработаны детальные рекомендации по модификации интерфейса с приоритизацией и roadmap внедрения.

**Структура рекомендаций:**

**Критические исправления (до запуска):**
- Переработать механизм фильтрации с внедрением мгновенного применения
- Добавить кнопку "Создать команду" в header раздела
- Оптимизировать формы управления пользователями для мобильных (multi-step approach)

**Высокоприоритетные улучшения:**
- Разработать interactive onboarding для новых пользователей
- Оптимизировать мобильные формы (bottom sheets, адаптивная высота)
- Внедрить систему обратной связи (loading states, notifications, validation)
- Создать Design System для обеспечения консистентности

**Среднеприоритетные улучшения:**
- Реструктуризация навигации раздела "Команда"
- GOMS-оптимизация форм (keyboard shortcuts)
- Улучшение мобильной аналитики
- Глобальный поиск
- Keyboard accessibility

**Roadmap внедрения:**
- Фаза 1 (2 недели): критические блокеры P0
- Фаза 2 (2 недели): высокоприоритетные P1
- Фаза 3 (3 недели): среднеприоритетные P2

**Прогноз улучшений:**
- SUS: 63.75 → 78-82 (+14-18 баллов)
- Usability веб: 67.9% → 82-85%
- Usability мобильное: 50% → 75-80%
- NPS: -25% → +35-45%
- SUS новичков: 35 → 62-70

**Документация:** `11_recommendations/interface_recommendations.md`

## Общие выводы

**Текущее состояние:**
Прототип ProjectFlow демонстрирует хорошее соответствие стандартам (90%+ по чеклистам) и эффективность базовых операций, однако не готов к запуску в текущем виде.

**Основные проблемы:**
- Критические функции (фильтрация, администрирование) имеют низкую успешность использования
- Мобильная версия значительно уступает веб-версии по всем метрикам
- Высокий порог входа для новых пользователей
- SUS балл ниже среднего по отрасли

**Рекомендации:**
Не рекомендуется запуск продукта до устранения критических проблем приоритета P0. Минимальный срок доработки: 4 недели.

**Ожидаемые результаты после внедрения рекомендаций:**
- Повышение SUS до уровня "хорошо-отлично" (78-82 балла)
- Достижение целевой успешности задач 80%+ на обеих платформах
- Переход NPS в положительную зону
- Снижение порога входа для новых пользователей
